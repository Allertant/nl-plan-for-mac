# Project Instructions

## Git Rules

- 在用户没有明确要求“提交”之前，不要主动执行 `git add`、`git commit`、`git push`。
- 完成功能实现、修复或重构后，默认停留在代码已修改但未提交的状态，等待用户确认。
- 如果用户只要求“修改代码”“修复问题”“实现功能”，不要将其推断为允许提交 Git。

## Documentation Rules

- 当任务涉及库、框架、SDK、API、CLI 工具或云服务时，优先使用 Context7 查询当前文档，不要仅凭记忆回答或直接编码。
- 包括但不限于：SwiftUI、SwiftData、AppKit、DeepSeek API、GitHub API、命令行工具、配置项、版本差异、迁移说明。
- 当用户询问 API 用法、配置方式、版本兼容、报错处理或最佳实践时，应先查 Context7，再基于查到的文档给出方案或实现。

## Communication Rules

- 当思考后认为自己没有准确理解用户需求，或存在会影响实现方向的关键歧义时，先向用户提问确认，不要直接按猜测执行。
- 如果可以通过阅读当前代码、文档或上下文自行消除疑问，应先自行确认；只有在关键需求仍不明确时再提问。
- 提问应简洁、直接，聚焦真正影响实现的问题，避免把本可以自己判断的细节转交给用户。
