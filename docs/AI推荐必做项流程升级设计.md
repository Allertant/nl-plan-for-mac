# AI 推荐必做项流程升级设计

## 1. 背景

当前系统已经具备 AI 推荐必做项的基础能力，但这套能力对“普通想法”和“项目”实际上采用了近似相同的输入方式：

- 普通想法：标题、分类、预估时间、状态、是否尝试过
- 项目：标题、分类、项目描述、规划背景

这在普通想法场景下基本成立，因为普通想法通常没有“长期进度”这一层语义，AI 只要知道：

- 这件事是什么
- 预估要多久
- 今天适不适合做

通常就足够做出合理推荐。

但在项目场景下，这套输入明显不够。项目不是一个静态条目，而是一个“持续推进中的对象”。如果 AI 只知道项目背景，不知道项目已经推进到了哪里，就会出现以下问题：

1. AI 每次都像第一次看到这个项目一样重新理解它。
2. AI 可能重复推荐已经做过的方向。
3. AI 很难判断当前阶段最合理的下一步。
4. AI 无法识别当前项目是在启动期、推进期、卡住期，还是收尾期。

因此，后续的 AI 推荐流程需要从“统一输入普通想法和项目”升级为“区分普通想法推荐与项目推荐的两套输入逻辑”。

---

## 2. 当前流程的人话描述

当前流程大致如下：

1. 系统从想法池中取出可推荐的候选项。
2. 系统再取出今天已经存在的必做项。
3. 把候选项和已有必做项一起组织成 prompt。
4. 发给 AI，请 AI 推荐 1 到 3 个适合今天做的事项。
5. 若推荐的是普通想法，则直接提升为必做项。
6. 若推荐的是项目，则允许 AI 生成一个项目切片任务。

当前输入给 AI 的数据主要包括：

### 2.1 想法池候选项

- `id`
- 标题
- 分类
- 预估时间
- 是否尝试过
- 当前状态
- 是否项目
- 项目描述
- 规划背景

### 2.2 今日已有必做项

- `id`
- 标题
- 分类
- 预估时间
- 当前状态

### 2.3 推荐策略

当前已有两种模式：

- 快速
- 挑战

---

## 3. 当前问题

### 3.1 普通想法没有大问题

普通想法通常没有“推进到哪里”的复杂语义，因此当前输入已经基本够用。

AI 需要做的主要是：

- 判断今天做不做
- 是否容易完成
- 是否与已有必做项冲突
- 是否适合当前剩余时间

### 3.2 项目存在本质缺口

项目推荐不只是“这个项目是什么”，而是“这个项目现在推进到了哪里，下一步最该推进什么”。

如果只给 AI：

- 项目标题
- 项目描述
- 规划背景

那么 AI 只能知道项目背景，却不知道：

- 已经完成了哪些推进任务
- 当前还有哪些未完成推进任务
- 最近是否遇到了阻塞
- 最近备注里是否改变了方向
- 当前项目正处于哪个阶段

这会导致项目推荐非常容易失真。

### 3.3 项目和普通想法不是同一个推荐问题

普通想法推荐更像：

**今天从候选项中选哪件做。**

项目推荐更像：

**这个长期项目的下一步最适合推进哪一个小切片。**

两者的输入结构和推理重点不应该完全一样。

---

## 4. 已确认的总体方向

### 4.1 引入“项目状态摘要”

项目在推荐前，不应直接把原始上下文全部发给推荐 AI，而应该先生成一份面向推荐的压缩摘要。

这份摘要的目标不是取代：

- 规划背景
- 项目备注
- 已完成/未完成绑定任务

而是将这些原始事实压缩成一段更适合推荐阶段消费的信息。

### 4.2 推荐分两类模式

后续推荐模式从“快速 / 挑战”调整为：

- `快速`
- `综合`

#### 快速模式

目标：

- 优先清理想法池
- 优先推荐易完成的普通想法
- 尽量减少 AI 输入成本和响应时间

策略：

- 默认优先只考虑普通想法
- 通常不进入项目摘要压缩流程
- 项目只有在普通想法候选明显不足时，才可作为补位项参与

#### 综合模式

目标：

- 综合考虑今天最值得推进的内容
- 普通想法和项目都可参与
- 不偏袒普通想法，也不偏袒项目

策略：

- 普通想法直接参与推荐
- 项目在参与推荐前，先准备最新可用的项目状态摘要
- 推荐 AI 基于普通想法 + 项目摘要统一排序

---

## 5. 项目推荐需要的原始数据

项目推荐需要读取两类上下文。

### 5.1 静态背景

解决“这个项目是什么”的问题。

建议包括：

- 项目标题
- 分类
- 项目描述
- 规划背景

### 5.2 动态进度

解决“这个项目做到哪里了”的问题。

建议包括：

- 已完成的绑定必做项
- 未完成的绑定必做项
- 最近项目备注
- 当前进度摘要
- 最近推进时间

### 5.3 动态数据的核心价值

这些数据不是为了让 AI 复盘历史，而是为了让 AI 理解：

- 当前阶段
- 最近推进内容
- 当前阻塞点
- 下一步最合理的方向

---

## 6. 项目状态摘要的定位

### 6.1 摘要不是长期事实

`规划背景` 是项目的相对稳定背景资料。  
`项目状态摘要` 不是背景，而是“面向推荐的一次性压缩视图”。

它本质上是：

- 对当前项目状态的压缩
- 为推荐阶段服务
- 未来可被重新生成和覆盖

### 6.2 摘要建议回答的问题

推荐阶段需要的项目状态摘要，建议至少回答这些问题：

1. 这个项目当前大致处于哪个阶段。
2. 最近已经推进了什么。
3. 当前还有哪些未完成推进项。
4. 当前是否存在明显阻塞。
5. 下一步最合理的推进方向是什么。

### 6.3 摘要建议的输出风格

建议是“短、稳、适合下游 prompt 消费”的自然语言摘要，而不是再返回一个很长的结构化大块文本。

例如可压缩成：

- 当前阶段
- 最近推进
- 当前阻塞
- 下一步建议方向

---

## 7. 摘要缓存与失效机制

### 7.1 不适合每次都重新生成

如果每次进入综合推荐都对所有项目现算摘要，会有两个问题：

1. 有些项目长期没有变化，重复计算浪费算力。
2. 候选项目多时，响应时间和 token 成本会明显上升。

因此摘要应当缓存，并带有失效判断机制。

### 7.2 核心思路

每个项目维护一个“推荐上下文更新时间”，同时摘要也记录它基于哪次上下文生成。

这样就能判断：

- 当前摘要还能不能用
- 是否必须重新生成

### 7.3 推荐的数据字段方向

建议为项目增加如下概念字段：

- `projectRecommendationContextUpdatedAt`
  - 表示所有“影响推荐的项目上下文”最后一次变化时间

- `projectRecommendationSummary`
  - 项目的推荐状态摘要正文

- `projectRecommendationSummaryGeneratedAt`
  - 摘要生成时间

- `projectRecommendationSummarySourceUpdatedAt`
  - 生成该摘要时所对应的项目推荐上下文更新时间快照

### 7.4 摘要可复用判断规则

当进入综合推荐时：

1. 如果摘要不存在，则必须生成。
2. 如果 `projectRecommendationSummarySourceUpdatedAt` 小于 `projectRecommendationContextUpdatedAt`，则摘要失效，必须重算。
3. 如果两者一致，则当前摘要可复用。

这比只比较“摘要生成时间”和“项目更新时间”更稳，因为它表达的是：

**这份摘要到底基于哪一版上下文生成。**

### 7.5 哪些变化会更新推荐上下文时间

不是所有字段变化都需要让摘要失效，只有会影响“下一步推荐应该做什么”的变化才算关键变化。

建议至少包括：

- 项目标题变化
- 项目分类变化
- 项目描述变化
- 规划背景变化
- 项目备注新增
- 项目备注修改
- 绑定必做项新增
- 绑定必做项解绑
- 绑定必做项标题变化
- 绑定必做项状态变化
- 绑定必做项备注变化
- 项目从普通想法改成项目
- 项目从项目改成普通想法

不应纳入的变化包括：

- 列表排序变化
- 展开/收起状态
- 与推荐无关的纯 UI 字段

否则会导致摘要频繁无意义失效。

## 8. 摘要生成时机

### 8.1 总体建议

摘要不建议作为每次推荐都强制现算的即时结果，也不建议在项目任何变化后立即后台重算。

更合理的是：

**推荐时按需生成。**

也就是：

1. 用户触发综合推荐。
2. 系统判断每个候选项目的摘要是否可复用。
3. 只对失效的项目重新生成摘要。
4. 生成完成后再统一进入推荐阶段。

这样能兼顾：

- 结果新鲜度
- 算力节省
- 架构复杂度可控

### 8.2 为什么不建议第一版做全量预生成

如果在每次项目备注变化、任务变更时就自动后台重算摘要，会带来额外复杂度：

- 需要维护后台调度
- 需要处理高频连续变更
- 需要处理失败重试
- 需要处理未完成写回与中断恢复

第一版没有必要引入这么重的链路。

## 9. 多项目场景下的并行摘要准备

### 9.1 需要并行

如果想法池中同时存在多个长期项目，而这些项目的摘要又都失效，那么串行生成摘要会明显拖慢推荐。

因此综合推荐前的摘要准备阶段，应支持并行。

### 9.2 不建议无限并发

虽然需要并行，但不建议无脑全部同时发起 AI 请求。

更合理的是：

- 有限并发
- 限制最大并发数，例如 2 到 3 个

原因：

- 避免 API 限流
- 避免整体失败率升高
- 避免一次性打满 token 消耗

### 9.3 推荐前的完整顺序

综合推荐推荐前的流程建议是：

1. 先筛出候选项。
2. 找出其中所有项目型候选项。
3. 判断哪些项目摘要失效。
4. 对失效项目做限流并行摘要生成。
5. 成功的写回摘要缓存。
6. 失败的走降级策略。
7. 汇总普通想法 + 项目摘要。
8. 再统一发起推荐请求。

### 9.4 失败时不应阻塞整轮推荐

如果某个项目摘要始终失败，不建议整轮推荐直接失败。

建议采用如下降级策略：

1. 若该项目已有旧摘要且未确认完全失效，可按策略决定是否临时降级复用。
2. 若没有可用摘要，则退化为只使用该项目的静态背景参与推荐。
3. 在推荐内部标记该项目上下文不完整。
4. 其他项目仍继续参与本轮推荐。

也就是说：

**摘要准备失败不应拖死整个推荐流程。**

## 10. 重试机制基础组件化

### 10.1 为什么需要抽取

摘要生成不是唯一会遇到 AI 调用失败的地方。系统中后续很多能力都可能需要统一的重试、超时与限流机制，例如：

- 项目状态摘要生成
- 项目进度分析
- 规划背景提示词生成
- 推荐必做项
- 其他 AI 结构化生成

如果每个流程都各自实现一套：

- 重试次数
- 退避时间
- 超时
- 错误分类

后续会越来越散，不利于统一维护。

### 10.2 推荐方向

建议抽取一个基础 AI 执行组件，统一承担：

- 最大重试次数
- 固定退避或指数退避
- 超时控制
- 并发限制
- 错误是否可重试判断
- 最终失败后的包装错误

上层业务只关心：

- 本次要执行什么 AI 任务
- 成功后如何消费结果
- 失败后如何降级

### 10.3 适用场景

该基础组件后续可直接服务于：

- 项目摘要准备阶段
- 综合推荐阶段
- 项目分析与规划相关 AI 能力

## 11. 推荐模式重构

### 11.1 当前模式的问题

当前已有的：

- 快速
- 挑战

更像是“按任务时长偏好排序”，而不是“按推荐目标选择流程”。

但实际上，推荐模式应该首先表达：

- 这一轮推荐是为了什么
- 要不要把项目纳入重型上下文流程

### 11.2 新模式建议

建议改为：

- `快速`
- `综合`

### 11.3 快速模式定义

目标：

- 主要清理想法池
- 快速形成完成感
- 降低输入 token 数量
- 缩短推荐响应时间

推荐对象：

- 优先普通想法
- 默认不走项目摘要压缩
- 项目仅在普通想法不足时作为补位

输入数据建议：

- 普通想法候选项
- 今日已有必做项
- 当前剩余时间
- 不包含项目摘要准备步骤

### 11.4 综合模式定义

目标：

- 在普通想法和项目之间做综合平衡
- 找出今天最值得推进的内容

推荐对象：

- 普通想法
- 项目型想法

输入数据建议：

- 普通想法候选项
- 今日已有必做项
- 项目静态背景
- 项目最新可用状态摘要
- 当前剩余时间

### 11.5 两种模式带来的收益

快速模式的收益：

- 无需准备项目摘要
- 大幅减少 token 消耗
- 更适合“今天先快速清掉一些事情”的场景

综合模式的收益：

- 项目也能被真正理解和公平参与
- 更适合“今天要认真安排整体推进重点”的场景

## 12. 推荐链路的目标升级

升级后的推荐链路不应再只是：

**从候选项里选几个。**

而应拆成两种思维模型：

### 12.1 普通想法推荐

目标：

- 选出今天最适合做的普通事项

AI 主要关注：

- 当前空余时间
- 任务完成成本
- 分类分散度
- 是否尝试过

### 12.2 项目推荐

目标：

- 找到长期项目下一步最值得推进的一个小切片

AI 主要关注：

- 当前阶段
- 最近推进
- 当前阻塞
- 未完成推进项
- 最合理的下一步方向

这两者最终仍可以统一汇总到一次推荐结果中，但前置准备过程不应一样。

## 13. 分阶段实施计划

以下阶段按推荐优先级排序。每一阶段都应在完成后明确标注状态，避免设计文档和真实代码状态脱节。

### 阶段一：推荐模式重构（已完成）

目标：

- 将当前推荐模式由“快速 / 挑战”调整为“快速 / 综合”。
- 明确两种模式的产品目标和 AI 输入差异。

范围：

- 调整推荐模式命名
- 修改模式文案
- 调整 prompt 语义
- 快速模式默认不走项目摘要链路

完成标志：

- 推荐 UI 中只保留“快速 / 综合”
- 快速模式不再触发项目摘要准备
- 综合模式保留项目参与能力

完成记录：

- 已完成：推荐模式由“快速 / 挑战”调整为“快速 / 综合”。
- 已完成：快速模式优先只向 AI 发送普通想法候选；当普通想法为空时，才退回到全部候选，避免推荐直接失效。
- 已完成：推荐 prompt 改为基于“快速清理想法池”与“综合权衡普通想法和项目”的语义，而不再使用“高难度优先”的旧策略。

### 阶段二：项目推荐上下文版本机制（部分完成）

目标：

- 为项目引入推荐上下文更新时间
- 为项目摘要引入基于哪版上下文生成的元数据

范围：

- 增加推荐上下文更新时间字段
- 增加摘要文本与摘要元数据字段
- 明确哪些业务事件会更新推荐上下文时间

完成标志：

- 项目具有推荐上下文失效判断能力
- 摘要可判断“可复用 / 需重算”

完成记录：

- 已完成：为项目补充推荐上下文更新时间字段 `projectRecommendationContextUpdatedAt`。
- 已完成：为项目补充推荐摘要与摘要元数据字段，包括摘要正文、生成时间、基于哪次上下文生成的时间快照。
- 已完成：项目读取时会补齐旧数据缺失的推荐上下文时间，避免历史项目一直处于无版本状态。
- 已完成：项目自身编辑、项目描述/规划背景更新、项目备注更新、绑定必做项新增/解绑、项目任务完成等关键操作会更新项目推荐上下文时间。
- 未完成：结算归档、项目相关必做项备注变化等次级链路仍未全部接入推荐上下文更新时间。

### 阶段三：项目状态摘要生成链路（已完成）

目标：

- 为项目生成推荐用状态摘要
- 将原始项目上下文压缩为适合推荐阶段消费的文本

范围：

- 定义摘要 prompt
- 定义摘要输入数据结构
- 生成结果写回数据库

完成标志：

## 14. 阶段完成后的标注规则

为避免“文档写了很多，但无法判断做到了哪里”，后续每完成一个阶段，都必须在本文件中做标注。

统一规则如下：

### 14.1 阶段标题标注

阶段标题统一使用以下三种状态之一：

- `（未开始）`
- `（部分完成）`
- `（已完成）`

示例：

- `### 阶段三：项目状态摘要生成链路（已完成）`

### 14.2 完成记录写法

每个阶段完成后，在该阶段下补充：

- 提交哈希
- 主要完成内容
- 尚未完成内容（若是部分完成）

推荐格式：

```md
完成记录：

- 提交：`abc1234 Commit message`
- 已完成：新增项目推荐摘要字段与失效判断逻辑
- 已完成：综合推荐前可识别摘要是否过期
- 未完成：尚未接入并行摘要生成
```

### 14.3 阶段推进原则

1. 不要求所有阶段一次完成。
2. 可以按阶段逐步落地。
3. 只要代码已经合入并验证可用，就应同步更新本文档状态。
4. 若阶段只完成部分能力，必须明确写“部分完成”，不能模糊写成“已完成”。

## 15. 当前建议结论

当前最推荐的落地方向如下：

1. 先把推荐模式改为“快速 / 综合”。
2. 快速模式优先推荐普通想法，不走项目摘要准备。
3. 综合模式才允许项目参与完整推荐。
4. 为项目引入推荐上下文版本机制。
5. 为项目引入可缓存、可失效的推荐状态摘要。
6. 综合推荐前，对失效项目做按需摘要准备。
7. 多项目场景下采用限流并行。
8. 重试机制后续抽成基础服务组件。

一句话概括这次设计升级的核心原则：

**普通想法推荐关注“今天选哪个做”，项目推荐关注“长期项目下一步推进哪一小步”，因此两者不能继续共用完全相同的输入模型。**
